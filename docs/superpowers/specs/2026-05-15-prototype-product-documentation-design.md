# 原型配套产品说明文档 — 设计规范

**状态**：评审通过（2026-05-15 第二轮 spec review：Approved）  
**日期**：2026-05-15  
**范围**：在现有交互原型（`src/` React 应用及/或 `legacy/` 静态原型）之上，补充**页面级说明**与**业务流程**，服务**混合读者**（研发/测试 + 业务/运营/管理层），且**维护人主要为 PM 单人**。

---

## 1. 背景与目标

### 1.1 现状

- 仓库内已有可交互原型（管理端 `/admin`、租户端 `/client` 等多路由页面）。
- **尚无**与之对齐的文字说明；业务规则、操作路径、mock 约定未成体系记录。

### 1.2 目标

- 在**不替换原型**的前提下，用**可维护**的方式把系统「讲清楚」。
- 同一套材料：业务侧能顺着**场景与故事线**理解；研发/测试能对照**路由与验收要点**。
- 控制单人维护成本，避免双写与文档腐烂。

### 1.3 非目标

- 不在本规范中规定具体业务文案、场景列表（由 PM 在落地时填写）。
- 不要求一次性覆盖全部页面；允许增量补齐。
- 不强制实现「原型内嵌说明层」；仅作为可选增强。

---

## 2. 方案结论（已确认）

采用 **「分层单手册为主 + 可选薄验收清单 + 原型内嵌说明作局部加分」**。

| 层次 | 内容 | 读者 |
|------|------|------|
| 主手册 | 场景优先 + 按需单页说明（固定小节模板） | 全员；业务多看前几节 |
| 薄验收清单（可选） | 从场景/单页第 6 节压缩的勾选表 | 测试、研发 |
| 内嵌说明（可选） | 仅 3～5 个关键屏或高频演示页 | 演示与对齐界面 |

### 2.1 主手册信息结构

**端到端场景文档**（优先建设 3～5 条）每篇固定结构：

1. 角色与目标  
2. 前置条件  
3. 点击路径（写明菜单与 **路由 path**，与 `src/App.tsx` 对齐）  
4. 业务结果与口径  
5. 与 mock / 假数据相关的一句话说明（如有）  
6. （可选）验收要点 — 与单页第 6 节同粒度；**默认不写**，仅当本场景需独立走查、且不拆到单页文档时填写

**单页说明**（按需增量）每篇建议小节（日常维护 **强制前 4 节**；5～6 节按需）：

1. 一页一句话（价值与边界）  
2. 角色与入口（端、菜单、路由）  
3. 主流程（3～7 步编号）  
4. 业务规则与状态（含关键术语）  
5. 数据与 mock 约定（研发/测试）  
6. 验收要点（可勾选）

跨页流程在场景文中只写跳转序列，**细节链到**单页小节，避免重复叙述。

### 2.2 交付与真源

- **权威真源**：仓库内 Markdown，建议目录 `docs/product/`（与代码同 PR 演进）。
- **入口文件**：`docs/product/README.md` 必须包含：如何启动/访问原型、文档阅读顺序、**与原型对齐的版本锚点**（如最近大改的日期、分支或 commit，由维护人更新）。
- **对外快照**：需要汇报或外发时，从 Markdown **导出** PDF/HTML；快照非日常唯一真源。
- **语雀/飞书等**：若团队需要，可采用 **单向同步**（从仓库导出或粘贴），并在文档头注明「以仓库某 commit/tag 为准」。

### 2.3 可选薄验收清单

- 从手册中已写定的「验收要点」汇总为表格（来源为 **单页第 6 节** 和/或 **场景第 6 节（可选）**；场景未写第 6 节时，以单页与汇总表为准）。表格字段建议：场景 ID、步骤、预期、关联路由。
- **不单独维护**第二套叙述性文字，避免与手册分叉。

### 2.4 可选原型内嵌说明

- 仅对**反复讲不清**或**演示高频**的少量页面考虑（例如侧栏说明、`?guide=1`）。
- 范围上限建议：**3～5 个关键路径或页面**；其余依赖手册。

---

## 3. 单人维护工作法

### 3.1 建设顺序

1. 先写 **3～5 条端到端场景**（覆盖最常讲、最易被问清的故事线）。  
2. 仅为场景中**复杂或争议**页面补充 `pages/` 单页说明。  
3. 有测试/研发稳定需求后，再生成或维护 **验收表**。

### 3.2 与原型变更同步

- 原型交互或路由变更：更新对应**场景段**及**受影响单页**；无单页文件时，优先在场景内更新路径说明。  
- 对外大版本：更新 `README.md` 中的版本锚点；必要时导出新的 PDF/HTML 快照。

### 3.3 止损原则

- 同一事实**只写一处**；场景不写重复字段说明。  
- 不追求首轮全量页面文档。  
- 内嵌说明默认不做，除非手册仍无法解决演示问题。

---

## 4. 建议目录结构

```text
docs/product/
  README.md              # 原型入口、阅读顺序、版本锚点
  scenarios/
    S01-<slug>.md        # 端到端场景（优先）
    ...
  pages/
    <optional>.md        # 与路由/菜单可辨识的 slug 命名
```

路由引用须与仓库内**实际路由定义**一致（当前主入口为 `src/App.tsx`；若后续拆分到独立路由模块，以该模块为准），便于检索与 diff。

---

## 5. 成功标准

- 新读者仅凭 **README + 场景文档** 可在原型上走完至少一条完整故事线并理解业务结果。  
- 研发/测试在需要验收时，能在 **单页第 6 节**、**场景第 6 节（若已撰写）** 或 **薄验收清单** 之一中找到可执行的验收表述（三者不要求同时存在，但不得互相矛盾）。  
- 单人维护时，单次原型小改对应的文档改动可定位在 **1～2 个文件** 内完成。
