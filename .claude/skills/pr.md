# /pr - 代码提交与推送流程

## 执行流程

按照以下步骤执行，每一步完成后才进入下一步：

### 第 1 步：收集变更信息

并行执行以下命令：
- `git status` — 查看当前工作区状态
- `git diff --stat` — 查看未暂存改动
- `git diff --cached --stat` — 查看已暂存改动
- `git log --oneline -5` — 查看最近提交记录

### 第 2 步：分析改动内容

根据 diff 结果，整理并输出以下信息（使用表格格式）：

**改动文件列表：**

| 改动类型 | 文件路径 | 说明 |
|---------|---------|------|
| 新增/修改/删除 | 具体路径 | 一句话描述 |

**改动行数统计：** +X 行 / -Y 行

### 第 3 步：影响范围评估

输出以下评估：

- **路由层**：路由是否有改动，是否影响已有菜单/页面访问
- **国际化**：是否有新增/修改的 i18n key
- **服务层**：是否有新增/修改接口，是否使用 Mock 数据
- **页面层**：新增还是修改已有页面，是否影响现有功能
- **风险点**：需要人工确认的风险项（如删除文件、重命名、破坏性改动等）

### 第 4 步：确认提交

将以上分析结果展示给用户，等待用户确认后再执行提交。

确认后执行：
1. `git add` 暂存所有相关文件（排除 `.env`、`credentials` 等敏感文件）
2. 生成符合项目规范的 commit message：
   ```
   <type>(<scope>): <subject> #SWASADMIN-4742

   <body>
   ```
   - type: feat/fix/docs/style/refactor/perf/test/chore
   - scope: 影响的模块
   - subject: 简短描述
   - body: 详细改动点（如有必要）

   最终追加：
   ```
   Co-Authored-By: Claude Opus 4.7 <noreply@anthropic.com>
   ```

3. `git commit` 提交代码
4. `git push` 推送到当前远程分支

### 注意事项

- 不要创建 PR（Pull Request），只需要提交并推送
- 如果工作区没有改动，直接告知用户"无文件需要提交"
- 如果暂存区已有文件，跳过 `git add` 步骤
- 敏感文件（.env、credentials、密钥文件等）不要提交，发现时提醒用户
